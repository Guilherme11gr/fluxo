Norte arquitetural

Runner como plano de execução verificável

O Project Manager organiza trabalho, prioridade e workflow. O runner executa em worktree, coleta evidência objetiva e devolve uma transição validada. A fronteira crítica é simples: nenhum agente, nem mesmo um orquestrador de feature, deve conseguir transformar narrativa em sucesso sem protocolo, artefato e destino de workflow.

Regra de ouro Para task de escrita, SUCCESS só existe quando a execução dona da task produziu delta git durável e escolheu uma disposição de workflow válida.
Modelo alvo

Três verdades, uma decisão

1. Protocol truth

A execução atual é dona do lifecycle. Exit code, texto final e processo filho não são autoridade suficiente.

2. Artifact truth

O git precisa provar entrega: baseline, HEAD final, commits novos e delta verificável.

3. Workflow truth

A task termina em destino válido: review, QA, done, blocked ou continuação explícita.

Topologia

Quem é dono de cada coisa

A arquitetura fica mais fácil quando o web app é tratado como control plane e o runner como execution plane. O erro perigoso é deixar o execution plane decidir sozinho que uma task terminou, ou deixar o control plane aceitar conclusão sem evidência.

Project Manager / Web

Fonte de workflow Projeto, feature, task, status, assignee, stage e próximo dono.
Agent API claim-next, heartbeat, events, comments e /finalize.
Policy guard Valida quem pode mover estado, com qual execution e com qual evidência.

Runner / Execution Plane

Claim + lease Recebe task, execution, runtime binding, memória e execução anterior.
Worktree por execução Isola branch, HEAD inicial, alterações e commit workflow.
Executor adapter OpenCode, Claude, Codex ou outro agente; produz logs e output contract.

Evidence / Audit

Git snapshot baselineHeadSha, finalHeadSha, arquivos e commits novos.
Run facts Exit code, duração, checks, stdout/stderr excerpts, eventos e logs completos.
Handoff estruturado Resumo humano + fatos objetivos persistidos na execution, não só em comentário.
Orquestração

Feature goal run, não empresa infinita

A unidade mais estável para autonomia não parece ser “um time de agentes trabalhando para sempre”. Parece ser uma run limitada por feature: ela planeja, dispara task runs, observa evidência e para quando a definição de pronto da feature foi atingida ou quando precisa de humano.

CEO agent como maestro de goal

O nome “CEO” funciona como metáfora operacional, mas a permissão dele precisa ser estreita: ele coordena a feature, não escreve código sem trilha, não aprova QA sozinho e não fica criando trabalho novo para parecer produtivo.

Limite saudável Toda feature run nasce com escopo, budget, critérios de parada, política de risco e definition of done. Quando faltar evidência, ela bloqueia ou pede humano.

Feature goal

FeatureRun Run de orquestração para uma feature específica, com deadline, budget e outcome esperado.
CEO / Orchestrator Quebra trabalho, escolhe próximo agente, interpreta bloqueios e mantém a feature coerente.
Stop conditions Sem evidência, loop repetido, risco alto, escopo mudando ou custo estourado.

Task attempts

Builder run Executa uma task de escrita e só entrega com artifact truth.
Reviewer run Revisa contra evidência do builder e política de stage.
QA browser run Executa QA real via Chrome DevTools MCP ou browser automation, com screenshots, traces e resultado.
Recovery run Repara disposição faltante e inconsistências, sem fingir entrega.

Feature disposition

Blocked Precisa de humano, credencial, ambiente ou decisão de produto.
Needs changes Retorna para task run com comentário estruturado e evidência do problema.
QA passed QA run anexa evidência navegável e autoriza transição para done.
Done Feature fechada só quando tasks críticas estão done e a definition of done passou.

Hardening

O contrato que fecha falso sucesso

O runner pode continuar usando texto do agente como explicação, mas não como autoridade. Autoridade vem de invariantes verificáveis.

Protocol truth

  • currentExecutionId fica preso na task enquanto a run está ativa.
  • expectedExecutionId é obrigatório em heartbeat, comment final, finalize e stage move.
  • Exit code 0 não encerra task; só uma transição terminal explícita encerra.
  • Run velha não pode fechar task depois de timeout, reclaim ou nova execução.

Artifact truth

  • baselineHeadSha aponta o HEAD antes da execução.
  • finalHeadSha aponta o HEAD validado no encerramento.
  • newCommitShas precisa conter commits gerados pela execução.
  • changedFiles é diagnóstico; sozinho não autoriza SUCCESS.
  • hasVerifiableDelta é o bit explícito para write tasks.

Workflow truth

  • Builder pode pedir review, bloquear ou continuar; não inventa aprovação.
  • Reviewer pode aprovar/rejeitar, mas não inventa evidência ausente.
  • System recovery corrige disposição faltante, não substitui entrega.
  • Task sem destino vira inconsistente e entra em recovery.

Recovery truth

  • successful run missing disposition vira watchdog, não silêncio.
  • Notice estruturado carrega source execution, evidência e ação requerida.
  • Circuit breaker bloqueia thrash depois de falhas repetidas.
  • Diagnostics apontam stale execution, missing evidence e reviewer path quebrado.
QA real

QA_READY como fila, não como quase done

O status atual já ajuda: QA_READY pode ser a fila que o QA runner claima. A mudança conceitual é que DONE deixa de ser o próximo passo automático e vira uma decisão produzida por uma QA execution com evidência.

Reviewer promove para QA Reviewer só envia para QA_READY quando existe evidência revisável: commits, arquivos, summary, riscos e instruções de validação.
QA runner faz claim Um agente com browser capability claima tasks em QA_READY e cria uma execution própria com expectedExecutionId.
Browser evidence Roda app, navega fluxos críticos, captura screenshots, console errors, network failures, traces e passos executados via Chrome DevTools MCP ou browser automation.
Passou: vira DONE Se os critérios passam, finalize usa nextStatus=DONE e anexa evidence pack da QA run.
Falhou: volta com handoff Se falha, a task volta para REVIEW ou DOING com bug notes, screenshots, repro steps e owner sugerido.

Evidence pack de QA

  • urlUnderTest, browser, viewport e ambiente.
  • stepsRun com resultado por etapa.
  • screenshots ou trace refs para auditoria humana.
  • consoleErrors e network failures categorizados.
  • acceptanceCriteria mapeados para pass/fail.

Handoff quando falha

  • Não marcar DONE quando QA detecta regressão.
  • Voltar para REVIEW quando a dúvida é de qualidade/critério.
  • Voltar para DOING quando há bug reproduzível de implementação.
  • Bloquear quando falta ambiente, credencial ou decisão de produto.

Matriz

Decisão por tipo de execução

primeiro corte de policy
Tipo Protocol truth Artifact truth Workflow truth Resultado permitido
Feature orchestrator FeatureRun atual, budget válido e plano de task runs auditável. Não produz artefato diretamente; exige evidência das task runs que coordena. Pode criar/ordenar runs e bloquear feature, mas não pular review/QA. Coordenação limitada por goal, não autonomia contínua sem critério de parada.
Builder write Execution atual, lease válido, terminal transition explícita. hasVerifiableDelta=true com commits novos. Review, QA, blocked ou continuação explícita. SUCCESS só se as três verdades passam.
Reviewer Reviewer autorizado e execution atual. Evidência revisável herdada do builder; não precisa criar commit. Aprova, rejeita, pede changes ou bloqueia com razão. Não promove para QA sem evidência revisável.
QA browser runner Execution atual claimada a partir de QA_READY. Evidence pack de browser: screenshots, traces, console, network e critérios pass/fail. Passa para DONE, retorna para REVIEW/DOING ou bloqueia. DONE só depois de QA pass quando a feature exigir validação real.
Research / no_write Execution atual e conclusão explícita. Opcional; sources/output estruturado substituem delta git. Done, blocked ou follow-up/delegation explícito. Workflow truth basta quando a task não promete artefato git.
System recovery Run de recovery identificada e idempotente. Não cria prova de entrega; só aponta ausência ou recupera disposição. Define próximo dono, bloqueia ou reabre caminho válido. Corrige inconsistência, não mascara falso sucesso.
Referências

O que copiar, adaptar e manter

Paperclip e Hermes não são concorrentes do desenho; são duas lentes maduras sobre o mesmo problema. O FluXo deve absorver mecanismos, não importar a arquitetura inteira.

Paperclip

  • Copiar: signoff policy explícita e gates por papel.
  • Copiar: recovery para run bem-sucedida sem disposição válida.
  • Copiar: notices estruturados, run events, excerpts e logs completos.
  • Adaptar: wakeup coordinator, coalescing, reason tracking e session state.
  • Evitar: deixar workflow truth substituir artifact truth.

Hermes Kanban

  • Copiar: completion via protocolo, não por processo encerrado.
  • Copiar: current_run_id / expected_run_id.
  • Copiar: attempts first-class com outcome, summary, metadata e log.
  • Adaptar: circuit breaker e diagnostics operacionais.
  • Evitar: review-required como convenção textual e git evidence sem validação.

FluXo

  • Manter: Project Manager como control plane central.
  • Manter: claim-next atômico com task, execution, lease e contextos.
  • Endurecer: git-aware execution virar critério de sucesso.
  • Endurecer: reviewer ancorado em evidência, não narrativa.
  • Adicionar: protocol truth como camada anterior ao artifact gate.
Produto

Autonomia estável, não fábrica de IA slop

A arquitetura precisa favorecer runs menores, evidência forte e paradas explícitas. O sistema deve parecer mais um PM disciplinado com bons executores do que uma simulação de empresa que trabalha o tempo todo porque consegue.

01 Goal bounded Feature run tem escopo e fim. Sem objetivo claro, o orquestrador não ganha trabalho novo.
02 Evidence first Texto explica; evidência decide. Git, browser, checks e logs vencem narrativa bonita.
03 Human interrupt Ambiguidade de produto, risco alto e repetição de falha viram bloqueio visível, não loop.
04 Small loops Runs curtas e auditáveis ganham de mega-runs. Melhor várias tentativas legíveis do que uma epopeia opaca.
Implementação

Sequência sugerida

A ordem é deliberada: primeiro impedir conclusão indevida, depois enriquecer recovery e observabilidade. Recovery é excelente, mas não deve ser o primeiro freio.

01 Endurecer runner e /finalize Execution não vira sucesso só por exit code ou JSON plausível.
02 Adicionar ownership explícito currentExecutionId na task e expectedExecutionId nas mutações.
03 Fechar contrato git evidence Baseline, HEAD final, commits novos, delta verificável e motivo quando ausente.
04 Modelar feature runs Goal, budget, stop conditions e orquestrador limitado por feature.
05 Reviewer gate Reviewer aprova/rejeita contra evidência herdada e policy de stage.
06 QA browser runner QA_READY vira fila claimable; DONE exige evidence pack.
07 Watchdog e recovery Detecta successful run missing disposition e cria notice/action idempotente.
08 Observabilidade pesada Full logs, excerpts, run events, diagnostics, circuit breaker e review isolado.
Definição curta

Como explicar para o time

Arquitetura alvo O FluXo é um Project Manager com runs de feature e um runner git-aware. O web app guarda a verdade do workflow; o orquestrador coordena goals limitados por feature; o runner executa em isolamento e mede a verdade do artefato; o QA runner valida experiência real no browser; a API de finalize age como kernel, aceitando transições somente da execução dona e somente quando as evidências batem com a promessa da task.
