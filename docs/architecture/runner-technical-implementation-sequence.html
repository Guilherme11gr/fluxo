Implementacao ideal

Plano tecnico para sair de 88% e chegar em 100%

Sequencia de trabalho para transformar o hardening ja validado em PRD em uma operacao fechada: contrato v1 completo, QA factual, evidence auditavel, smoke e2e, PR policy, chaos tests, CI e release versionado do runner.

Regra principal

Nao avancar para features de operacao enquanto o kernel ainda depende de texto humano para provar que teste, typecheck, delta e Git policy foram realmente cumpridos.

Primeiro cortechecksRun factual + evidence canonica.
Segundo corteSmoke e2e automatizado contra FLXO em PRD.
Terceiro cortePR policy, chaos tests e UI operacional.
Principios

Como implementar sem perder o protocolo

Esses principios evitam que o runner volte a depender de convencao verbal do agent. Toda decisao importante precisa estar em payload, metadata, teste ou link verificavel.

Fato observado vence texto

Se o agent diz que testou, mas o runner nao observou comando com exit code zero, o sistema nao deve tratar como QA passada.

Finalize e o gate central

O runner pode enriquecer evidence, mas a API precisa continuar sendo a autoridade que valida owner, role, transition, artifact, QA e Git policy.

Compatibilidade progressiva

Ambientes antigos podem usar fallback, mas o caminho novo deve ser explicito: projectKey, JSON v1, evidence v2 e links Git.

Smoke e2e automatizado

Fluxo

  1. Ler AGENT_API_KEY de .env.local.
  2. Criar feature/task fake no FLXO.

PR slicing

Como dividir sem criar regressao

Mesmo trabalhando direto na main quando necessario, esta divisao ajuda a manter cada deploy pequeno o bastante para diagnosticar.

Corte A: QA factual

Apenas runner result/check collection + testes Go. Nao mexer em UI nem PR policy. Aceite: nova run tem checksRun.

Corte B: Evidence API

Normalizar e persistir evidence no finalize. Aceite: API retorna metadata.evidence com artifact/qa/git.

Corte C: Smoke script

Automatizar FLXO fake run e asserts. Aceite: script falha em exit non-zero quando algum contrato quebra.

Corte D: PR policy

Implementar e testar policy com PR obrigatorio. Aceite: sem PR rejeita, com PR avanca e publica link.

Corte E: UI + CI + release

Expor evidencia para operador, endurecer pipeline e versionar binarios. Aceite: release reproduzivel e CI bloqueando regressao.

Nao avancar se

Guardrails para a sequencia

01 Checks vazios

Se comandos foram executados e checksRun ainda fica vazio, nao faz sentido seguir para PR policy ou UI. O kernel ainda nao esta auditavel.

02 Evidence duplicada e divergente

Se metadata.git, metadata.result.git e metadata.evidence.git discordarem, parar e normalizar antes de nova run.

03 Smoke manual demais

Se a validacao exige copiar IDs no terminal, ainda nao e 95%. O smoke precisa gerar task, rodar, validar e imprimir links sozinho.
