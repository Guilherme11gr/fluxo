Roadmap para 95%

Fechar a auditabilidade

Prioridade alta

Contrato v1 completo

Exigir FLUXO_RESULT_JSON_START/END alem do summary. Se o agent omitir o JSON, o runner deve reparar com fatos observados ou marcar a saida como incompleta.

  • schemaVersion: "v1" obrigatorio.
  • checksRun nao pode ficar vazio quando comandos foram executados.
  • Teste cobrindo summary-only, JSON valido e JSON quebrado.

QA por fato observado

O runner precisa construir evidencia a partir dos comandos reais, nao apenas confiar na frase final do agent.

  • Capturar comando, exit code, duracao e status.
  • Rejeitar sucesso se uma checagem critica falhou.
  • Expor metadata.qa.passed e metadata.qa.checksRun.

Evidence schema

Consolidar a estrutura que o finalize recebe e persiste para auditoria independente do comentario.

  • artifact.changedFiles e hasVerifiableDelta.
  • git.policyVerified, branch, commit, compare e PR.
  • qa.required, qa.passed e detalhes.

Paridade da API em PRD

Hoje o runner tem fallback para buscar projectKey. Bom para compatibilidade, mas a API de claim deve retornar o campo diretamente.

  • claim-next devolve projectKey: "FLXO".
  • Backend e runner compartilham a mesma branch semantica.
  • Fallback continua para ambientes antigos.

Smoke e2e automatizado

O teste manual da FLXO-388 deve virar script repetivel para validar PRD sem lembrar todos os passos de cabeca.

  • Criar task fake no FLXO via API.
  • Rodar runner --once.
  • Validar status, git links, checks, cleanup e task em REVIEW.

Janitor local pos-crash

O server reap funcionou. Falta o runner limpar automaticamente restos locais de execucoes que ja terminaram ou expiraram no servidor.

  • Remover locks orfaos no startup.
  • Prunar worktrees de executions terminais.
  • Logar o que foi limpo sem bloquear a proxima run.

Sequencia que maximiza confianca

Proximo passo claro

Fechar checksRun e QA factual

Instrumentar o runner para transformar comandos observados em checks estruturados. Depois rodar outra task FLXO fake e exigir que metadata.result.checksRun venha preenchido.

Consolidar evidence/artifact no finalize

Tornar a estrutura de evidencia explicita no payload, na persistencia e nos testes de rota. O comentario continua humano; a metadata vira a fonte auditavel.

Publicar paridade da API em PRD

Subir projectKey no claim-next e manter fallback no runner. A branch semantica deixa de depender de chamada extra em producao.

Criar smoke e2e automatizado

Um script deve criar task, rodar runner, aguardar finalize e validar todos os campos criticos. Esse script vira o botao de confianca antes de mexer no protocolo.

Atacar PR policy e chaos tests

Quando o 95% estiver estavel, testar os casos que quebram operacao real: PR obrigatorio, runner concorrente, stale finalize, owner errado e patch indevido.

Definition of done

Como saber que chegou la

Done para 95%

  • Nova run FLXO termina em SUCCESS e task em REVIEW.
  • metadata.result.checksRun lista teste e typecheck executados.
  • metadata.git.links inclui repository, branch, compare e commit.
  • hasVerifiableDelta=true e newCommitShas nao vazio.
  • Crash anterior nao deixa lock/worktree impedindo nova run.

Done para 100%

  • Policy com PR obrigatorio validada em sucesso e rejeicao.
  • UI mostra toda evidencia sem abrir JSON bruto.
  • Chaos suite cobre ownership, stale execution e concorrencia.
  • CI roda os checks reais do app e do runner.
  • Runner tem binario versionado e reporta versao/commit no heartbeat.
