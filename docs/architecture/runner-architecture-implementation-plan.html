Plano técnico

Como transformar o runner em kernel de execução

Esta página aprofunda o mapa visual: o que existe hoje, quais invariantes faltam, quais campos e contratos entram, e em que ordem implementar para chegar em feature runs com QA real sem criar uma máquina autônoma opaca.

Resumo da tese

O FluXo não precisa virar “empresa de agentes”. Precisa virar um sistema de goals limitados: feature run coordena, task execution tenta, reviewer/QA validam, e /finalize decide com base em protocolo, artefato, workflow e evidência de QA.

Revisão

Estado atual do código

A base já tem quase todos os tijolos operacionais. A lacuna é que eles ainda não formam um kernel: claim-next cria a execução, o runner executa, mas /finalize ainda aceita a intenção final com pouca validação semântica.

Já existe

  • Task/Feature/Agent como modelo de trabalho e dono.
  • AgentExecution com status, output, metadata, heartbeat e eventos.
  • ExecutionLease por projeto, runner e execution.
  • RuntimeBinding com repo, branch base, git policy e execution mode.

Funciona hoje

  • claim-next filtra status e assignee, cria execution e lease.
  • runner-go prepara worktree, executa agente, cria comentário e chama finalize.
  • Agente configurável usa pick/claim/done status, então QA já cabe no modelo.
  • Events já têm tabela sequencial por execution.

Falta endurecer

  • Task.currentExecutionId não existe como lock semântico.
  • Git snapshot mistura HEAD herdado com commits novos.
  • Finalize não valida role, transition matrix nem evidence contract.
  • QA_READY ainda pode ser tratado como quase final, não fila de QA real.
Contratos

Quatro gates no finalize

O endpoint de finalize deve virar o ponto onde a intenção do agente encontra as provas do sistema. O payload pode continuar rico, mas a decisão final precisa ser computada pelo backend.

01 Parse request Zod valida forma, mas não decide sucesso.
02 Load context Execution, task, feature, agent, lease e runtime metadata.
03 Protocol gate Route id, expectedExecutionId, currentExecutionId e terminal idempotente.
04 Role gate Builder, reviewer, QA, orchestrator ou recovery são derivados do servidor.
05 Artifact gate Write task success exige baseline, final HEAD, commits novos, delta e branch policy.
06 QA gate DONE pós QA_READY exige browser evidence pack quando aplicável.
07 Persist atomically Execution, task, currentExecution clear, lease delete, event e comment; reject não aplica transição.
08 Wake next actor Reviewer, QA, recovery ou orchestrator da feature.
Payload

Finalize v2 recomendado

O objetivo não é abandonar o payload atual. É acrescentar campos que tornam a decisão verificável e reduzem inferência espalhada no backend.

function validateFinalize(ctx, req) {
	  assert(req.expectedExecutionId === ctx.execution.id)
	  assert(ctx.task.currentExecutionId === ctx.execution.id)
	  assert(ctx.execution.status in ["CLAIMED", "RUNNING"])

  const effectiveRole = deriveExecutionRole({
    execution: ctx.execution,
    agent: ctx.agent,
    task: ctx.task,
    runtimeBinding: ctx.runtimeBinding
  })
  // req.callerRoleHint is telemetry only. It never grants authority.
  assertTransitionAllowed(effectiveRole, ctx.task.status, req.disposition)

	  if (req.status === "SUCCESS" && requiresArtifact(ctx, effectiveRole)) {
	    assertGitPolicy(ctx.runtimeBinding, req.evidence.artifact)
	    assert(req.evidence.artifact.hasVerifiableDelta)
	    assert(req.evidence.artifact.newCommitShas.length > 0)
	    assert(req.evidence.artifact.baselineHeadSha)
	    assert(req.evidence.artifact.finalHeadSha)
	    assert(req.evidence.artifact.baselineHeadSha !== req.evidence.artifact.finalHeadSha)
	  }

	  if (req.disposition.blockReason) {
	    assertBlockOperationAllowed(effectiveRole, ctx.task.status, req.disposition)
	    assert(req.disposition.requestedNextStatus == null)
	  }

  if (movesToDoneFromQa(effectiveRole, ctx.task.status, req.disposition)) {
    assertQaEvidence(req.evidence.qa)
  }
}
Rejected finalize

Sem degradação silenciosa

Se o caller pede SUCCESS mas falha nos gates, o kernel não deve “rebaixar” como se fosse uma falha normal do agente. A intenção é rejeitada explicitamente, a transição pedida não é aplicada e a auditoria mostra qual invariant falhou.

MVP sem novo enum

  • Execution não é marcada terminal quando o finalize é rejeitado.
  • Event recebe finalize.validation_rejected com o gate que falhou.
  • Task não aplica requestedNextStatus.
  • Ownership permanece até retry válido, cancelamento explícito ou reap seguro.

Modelo alvo

  • VALIDATION_REJECTED para payload inválido ou evidence insuficiente.
  • INCONSISTENT para estado impossível ou divergência sistêmica.
  • FAILED fica reservado para execução que tentou e falhou.

Resposta da API

  • HTTP 409 para stale execution / ownership inválido.
  • HTTP 422 para evidence ou transition inválida.
  • Event finalize.validation_rejected sempre persistido quando possível.
Authority

Role nunca vem do caller

Esse é um ponto de segurança do desenho. O runner pode enviar um hint para debug, mas a autoridade real vem da execution persistida, do agent configurado e do contexto de claim.

Não fazer

  • Não confiar em req.role.
  • Não permitir que uma execution se declare QA/reviewer no finalize.
  • Não derivar permissão de texto final, prompt ou comentário.

Fonte de verdade

  • Agent.type/config define capabilities permitidas.
  • Execution.metadata.runRole é gravado pelo servidor no claim.
  • Task.status + claim config restringem o papel efetivo.

Uso do hint

  • callerRoleHint serve para telemetria e diagnóstico.
  • Divergência entre hint e effectiveRole vira warning/event.
  • Policy sempre usa effectiveRole.
Schema

Mudanças de dados

Dá para fazer em duas velocidades: primeiro metadata compatível para reduzir risco; depois campos normalizados onde o backend precisa consultar, bloquear e auditar com eficiência.

Camada Mudança mínima Mudança alvo Por quê
Task ownership metadata.currentExecutionId só como transição se a migration precisar esperar. Task.currentExecutionId com FK opcional para AgentExecution já no primeiro hardening PR. Impede finalize, heartbeat e stage move por execução velha.
Execution role execution.metadata.runRole gravado pelo servidor no claim. Coluna runRole indexada ou enum dedicado, nunca enviada como autoridade pelo caller. Permite transition matrix sem depender só do tipo do Agent e sem role spoofing.
Feature goal Feature.metadata.goalRun se existir campo JSON futuro. Novo modelo FeatureRun com status, budget, plan e outcome. Feature vira unidade de autonomia bounded, como Codex/Hermes goals.
Evidence Persistir em AgentExecution.metadata.evidence. Opcionalmente tabela ExecutionArtifact para arquivos, traces e screenshots. QA/browser e logs grandes não devem virar texto solto em comment.
Recovery Eventos system.notice e comments estruturados. Tabela/metadata para RecoveryAction idempotente. Watchdog precisa lembrar que já tentou corrigir uma inconsistência.

Artifact truth

Git snapshot v2

O runner precisa parar de usar uma lista genérica de commits como prova. Para escrita, a diferença entre baseline, HEAD final e commits novos é a peça que fecha o falso positivo.

Problema atual

  • CommitChanges retorna "" sem erro quando não há mudança.
  • CaptureGitSnapshot pode gravar o HEAD atual em commitShas.
  • Finalize não sabe diferenciar HEAD herdado de commit produzido pela run.

Contrato novo

  • baselineHeadSha capturado antes do executor.
  • finalHeadSha capturado depois de commit/PR.
  • newCommitShas coletado por range baseline..HEAD.
  • hasVerifiableDelta calculado pelo runner e revalidável pelo backend.
  • branch/policy validado contra runtime binding: branch base, prefixo permitido e branch protegida.
type GitEvidence struct {
  GitPolicy            string   `json:"gitPolicy"`
  BaseBranch           string   `json:"baseBranch"`
  Branch               string   `json:"branch"`
  BaselineHeadSHA      string   `json:"baselineHeadSha"`
  FinalHeadSHA         string   `json:"finalHeadSha"`
  NewCommitSHAs        []string `json:"newCommitShas"`
  ChangedFiles         []string `json:"changedFiles"`
  HasVerifiableDelta   bool     `json:"hasVerifiableDelta"`
  PolicyVerified       bool     `json:"policyVerified"`
  VerificationErrors   []string `json:"verificationErrors"`
}
// Rule of thumb:
if workKind == "write" && status == "SUCCESS" {
  require(gitEvidence.PolicyVerified)
  require(gitEvidence.GitPolicy == runtimeBinding.GitPolicy)
  require(gitEvidence.BaseBranch == runtimeBinding.DefaultBaseBranch)
  require(branchAllowed(gitEvidence.Branch, runtimeBinding.AllowedBranchPrefix))
  require(!isProtectedBranch(gitEvidence.Branch, gitEvidence.BaseBranch))
  require(gitEvidence.HasVerifiableDelta)
  require(len(gitEvidence.NewCommitSHAs) > 0)
  require(gitEvidence.BaselineHeadSHA != gitEvidence.FinalHeadSHA)
}

// changedFiles remains useful for diagnosis,
// but it is not enough for final SUCCESS.
QA runner

Como encaixar browser QA sem quebrar o board

A pergunta do handoff fica simples se QA_READY deixar de ser “pronto para done” e virar “pronto para ser claimado por QA”. O status depois é DONE somente quando a QA execution passa. Na primeira versão, a task permanece em QA_READY durante a execução; quem indica “QA em andamento” é currentExecutionId + execution status, não um novo status.

Configuração de agentes

Builder pick=TODO, claim=DOING, done=REVIEW.
Reviewer pick=REVIEW, claim=REVIEW, done=QA_READY, nextAssignee=QA.
QA browser pick=QA_READY, claim=QA_READY, done=DONE.

QA execution

Boot app Runner inicia dev server ou usa URL deploy/preview definida pelo runtime binding.
Browser tool Adapter usa Chrome DevTools MCP, Playwright ou browser-use via provider configurado.
Scenario pack Critérios vêm da task/feature, reviewer handoff e smoke suite padrão do projeto.
Evidence pack Passos, screenshots, traces, console errors, network failures e pass/fail por critério.

Disposição

Pass Finalize com nextStatus=DONE e evidência de QA anexada.
Ambíguo Volta para REVIEW com pergunta objetiva e artefatos.
Bug reproduzível Volta para DOING ou cria BUG vinculado, com repro steps.
Ambiente quebrado Bloqueia com razão: credencial, seed, URL, serviço externo ou setup.

QA pass rule:
  task.status == QA_READY
  effectiveRole == qa
  task.currentExecutionId == execution.id
  requestedNextStatus == DONE
  qa.passed == true
  qa.checks.length > 0
  no critical console/network failures

QA fail rule:
  effectiveRole == qa
  task.currentExecutionId == execution.id
  requestedNextStatus in [DOING, REVIEW]
  comment includes repro steps
  evidence.qa.passed == false
Feature run

CEO agent como orquestrador bounded

O CEO não deve ser “um chefe geral sempre acordado”. Ele deve ser uma execution de feature com objetivo e limites. Ele decide a próxima tentativa, mas não substitui builder, reviewer, QA nem finalize gates.

Responsabilidade Permitido Proibido Sinal de parada
Planning Quebrar feature em tasks pequenas, ordenar risco e escolher primeiro agente. Criar escopo novo fora da feature sem aprovação humana. Escopo ambíguo, critério de aceite ausente ou conflito de prioridade.
Coordination Reagir a finalize events e acordar builder/reviewer/QA/recovery. Marcar task ou feature como done por narrativa própria. Mesmo erro N vezes, custo estourado ou task sem dono válido.
Quality pressure Exigir checks, QA, screenshots e handoff estruturado. Diminuir gate para fazer a feature andar. QA bloqueado por ambiente, credencial ou dado não disponível.
Feature closure Propor encerramento quando todas as tasks críticas estão DONE e DoD passou. Fechar feature com tasks ativas, blocked ou sem QA exigido. Definition of done não verificável.

FeatureRun state machine

  • PLANNING: cria plano inicial e budgets.
  • RUNNING: coordena task runs e observa eventos.
  • WAITING: aguarda humano, ambiente ou dependência externa.
  • BLOCKED: não consegue avançar sem intervenção.
  • DONE: DoD passou e não há trabalho ativo relevante.

Budgets anti-slop

  • maxTaskRuns: limite de tentativas por feature.
  • maxRetriesPerTask: evita bater cabeça no mesmo bug.
  • maxWallClockMinutes: corta feature run longa demais.
  • maxOpenQuestions: força humano quando o contexto ficou difuso.
  • requiresQa: define quando browser QA é obrigatório.
Transition matrix

Quem pode mover o quê

Essa matriz é o ponto onde Paperclip entra mais forte. Não basta ter status; cada papel tem um conjunto pequeno de transições válidas.

Effective role From To permitido Evidência exigida Observação
orchestrator feature active assign task, create task, block feature, request human plan diff, reason, budget state Não finaliza task nem feature sozinho.
builder TODO/DOING REVIEW, block(reason), TODO retry git evidence para write; summary/checks para no_write Não deve mandar direto para DONE em write flow.
reviewer REVIEW QA_READY, DOING/TODO changes, block(reason) builder evidence + review findings Não inventa evidência quando builder não entregou.
qa QA_READY DONE, REVIEW, DOING, block(reason) browser evidence pack DONE é resultado de QA pass, não default.
recovery inconsistent block(reason), TODO/REVIEW/QA_READY with notice source execution, inconsistency reason Corrige workflow, não fabrica artifact truth.
API / Runner

Mudanças por componente

Agent API

  • claim-next seta current execution na task.
  • heartbeat valida expected execution e renova lease antes do evento.
  • finalize deriva effectiveRole e rejeita transições inválidas explicitamente.
  • reap-stale limpa current execution com compare-and-clear por execution id.
  • events recebem system notices e validation failures.

runner-go

  • GitEvidence v2 captura baseline antes e final depois.
  • NoChanges vira condição explícita, não sucesso silencioso.
  • Finalize v2 manda expectedExecutionId, callerRoleHint e evidence.
  • QA adapter roda browser provider e persiste artifacts.

Web / PM

  • FeatureRun view mostra goal, plan, attempts e stop conditions.
  • Task detail mostra evidence pack, não só comentário.
  • QA column deixa claro que QA_READY é fila ativa.
  • Recovery notices aparecem como ação requerida.
Rollout

Ordem para chegar lá sem quebrar tudo

Eu evitaria começar pelo CEO agent. Primeiro fecha protocol truth + artifact truth no mesmo corte de hardening; depois QA; depois feature run. Assim a camada de orquestração já nasce em cima de trilhos sólidos.

Fase 0 Configurar QA com o que já existe
  • Agent QA como CUSTOM ou tipo novo.
  • pickStatus=QA_READY, claimStatus=QA_READY, doneStatus=DONE.
  • Reviewer aponta nextAssigneeAgentId para QA.
  • Sem enforcement forte ainda, só prova de fluxo.
Fase 1 Protocol + artifact gate
  • currentExecutionId na task e set no claim.
  • expectedExecutionId em finalize, heartbeat, events, comments e stage move.
  • finalize/reap-stale limpam ownership somente quando ainda aponta para a execution.
  • effectiveRole gravado pelo servidor no claim.
  • GitEvidence v2 no runner.
  • QA runner envia evidence.qa mínimo; browser pack entra como upgrade.
  • hasVerifiableDelta obrigatório para write success.
  • Finalize rejeita falso success sem marcar execução como terminal.
  • Tests em runner-go e rotas agent críticas.
Fase 2 Protocol hardening completo
  • Enum/status explícito para VALIDATION_REJECTED se o produto quiser terminalizar rejeição.
  • Run stale recebe evento próprio quando tenta mutação fora do owner.
  • Lease cleanup consolida ownership de execuções órfãs e em estado inconsistente.
  • Transition matrix completa passa a diferenciar status change de block/unblock.
Fase 3 QA browser evidence
  • Adapter browser provider.
  • Evidence pack schema com screenshots/traces reais.
  • DONE continua exigindo QA pass quando effectiveRole=qa.
  • Falha volta com repro steps.
Fase 4 FeatureRun / CEO bounded
  • Modelo FeatureRun.
  • Orchestrator claim/wakeup por feature.
  • Budgets e stop conditions.
  • Event-driven next action após finalize.
Fase 5 Recovery watchdog
  • successful run missing disposition.
  • system notices idempotentes.
  • diagnostics para stale/missing evidence.
  • circuit breaker por falhas repetidas.
Fase 6 Observabilidade pesada
  • Full logs + excerpts.
  • Artifact refs para QA e traces.
  • FeatureRun dashboard.
  • Review isolado para PR não confiável.
Fase 7 Policy hardening
  • Transition matrix declarativa.
  • Signoff policy por projeto.
  • Escalation rules para humano.
  • Métricas de quality drift.
Primeiro PR bom

Menor corte que muda o jogo

Eu começaria aqui: adicionar Task.currentExecutionId, gravar esse campo no claim-next, enviar expectedExecutionId no finalize, adicionar GitEvidence v2 no runner e fazer /finalize rejeitar SUCCESS de write task sem ownership válido, branch policy válida e hasVerifiableDelta. Isso fecha protocol truth e artifact truth antes de introduzir CEO agent, QA browser ou FeatureRun.

Depois desse corte, QA runner fica bem natural: é “só” mais um role claimando QA_READY, mas com evidence pack próprio.
